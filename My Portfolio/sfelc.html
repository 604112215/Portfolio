SFELC WEBDESIGN

SFELC WEBDESIGN is the first version of the website for San Francisco Engineering Leadership Community (SFELC), a community for engineering leaders in Silicon Valley. SFELC providing event, peer group, training program for our users to improve their soft skills( e.g. communication, self-awareness, etc.). We now have 2000+ leaders(includes CTO, VP) from over 700 companies.

My Role

I am the only designer in this team during my internship. I designed UI/UX independently for SFELC website including Sign in/up flow, RSVP & Payment flow, Book List & Comments section, etc. I worked very closely with the engineers, product manager, and founder. Created and maintained design guidelines and libraries which could be used by the whole design and engineering team.

RSVP & Payment Flow

RSVP shows all the information about the upcoming event. We are hosting two types of events: Charged Event and Free Event. However, there are some problems in user side waiting to be solved

Problem

Users want to know the number of left spots before making a purchase decision*

Solution

Visualize the number of left spots on tickets.

Problem

Users want to enroll in the Waiting List if there are no available spots. However, If we allow users to join the waiting list for charged events, the situation of charging users become complexed.*

Solution

Waiting List only works for Free Event.

Free Event
1. If there are available spots, the user can directly RSVP into the upcoming event and get a follow-up confirmation email.
2. If the event is free but doesn’t have available spots left, the user can Join Wait List. Once there are any open spots, they will enroll in the event automatically and get follow-up confirmation. (Join Wait List only available for a free event)

Charged Event
1. If there are available spots, the user needs to pay the event fee and then RSVP into the upcoming event and get a follow-up confirmation email.

Iteration

I did user testing when I was designing “BOOKLIST”. I got some feedback from users. Their feedback and reaction about my design helped me made the final design decisions. Also, There are some technical constraints that influence the design decision making.

After discussing with the PM and Engineering team, our team decided to apply Stripe API to build up the RSVP & PAYMENT FLOW. In order to make the UX design work on Stripe API, I have to learn UI components of it. So I designed the first version of RSVP & PAYMENT flow and discussed with PM and Engineers about it.

I discussed the UX content with PM and Founder(Jerry), and I decided to change some of UI contents in the previous design.
1.Remove the amount of tickets.
We only allow each of our member buy one ticket because it is a way to encourage more people to sign up our community.
2.Remove “Billing Address”.
I asked the founder and PM:” Do we need mail paper payment receipts or bills to our users?” The answer is “No.” We don’t mail user any paper bill or advertising. However, filling this “Billing address” make this flow hard to finish.

Recommend A Book-Content

To find the book that recommended by users, engineers need to know some basic information about the book.
I interviewed Engineers about what kind of information do they need to manage data. They said that the most efficient way to maintain this function is asking users providing “The name of the book,” “Category of this book,” “URL of this book,” ,and “Reason of recommendation.”
However, Providing such amount of information is a challenge for users. They may lose interest in recommending books. So I decided to remove the “ Category,” and turn the “Book URL” as an optional request, and let users feel free to recommend a book to us.

Comments Section

There is a question come to my mind when I was designing the comments section.

Problem

How many layers of reply do we support in the comments section?

Sometimes, there could have infinity layers of reply when user reply other people’s reply for a comment. This situation will influence how the experience look like. I turned to ask an engineer about this question.
Finally, We decide only support two layers of reply in the "comments section". Because this is an efficient way for engineer to manage those data, and it is also good for users to have a clear reply experience.
